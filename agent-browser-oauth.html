1. The reframe: your blocker is authentication, not reachability

OAuth has two hard parts for an agent, and they're worth separating:

  1. The human login step — password, 2FA, and getting past anti-bot detection (reCAPTCHA, “this browser may not be secure”). Providers actively fight automation here. This is ~95% of the pain.
  2. The redirect — the provider bounces the browser to http://localhost:PORT/callback?code=..., and the app swaps that code for a session.
Key insight

Tailscale only ever helps with reachability (making some URL resolve to your local app). Your problem is #1, authentication. So the real fix is to never make the agent log in at all — reuse a session you already established as a human.

The good news: agent-browser is built for exactly this.

2. The actual best fix: session reuse (no OAuth dance at all)

From agent-browser --help, you have these auth primitives:

--auto-connect       Connect to a running Chrome to reuse its auth state
                     Tip: agent-browser --auto-connect state save ./auth.json
--state <path>        Load saved auth state (cookies + storage) from JSON
--profile <path>      Persist login sessions across restarts (cookies, IndexedDB, cache)
--session-name <n>    Auto-save/restore cookies + localStorage by name

Two clean patterns, in order of preference:

A. Borrow your already-logged-in Chrome

You're already authenticated to Google in your daily browser. Capture that session once and replay it:

# grab live session from a running Chrome
agent-browser --auto-connect state save ./auth.json

# launch pre-authenticated — no login page ever appears
agent-browser --state ./auth.json open https://yourapp.localhost
Why this wins

The agent inherits a session token, never sees a login page, and never touches your password. The exact thing that makes Google login painful (anti-bot) is the thing you skip entirely.

B. Persistent profile — log in once, manually

Run headed, do the OAuth login yourself in that window the first time; the profile keeps you logged in forever after:

agent-browser --profile ~/.cache/agent-browser/myapp --headed open https://yourapp.localhost
# complete Google login in that same window once; later runs reuse it

Because it's the same browser that started and finished the flow, the callback completes naturally — nothing to copy. For both A and B, plain localhost is fine and no Tailscale is needed.

3. The Telegram link-handoff (your idea) — it works

Your instinct is correct, and it's a legitimate fallback for when you genuinely need a fresh interactive login each run. The precise mechanism:

  1. The agent's browser starts the flow (so it holds the state/PKCE binding and the app session cookie); you grab the auth URL.
  2. Agent sends you the accounts.google.com/... URL — this is literally your /ts Telegram skill (“send the link to me”).
  3. You open it on your phone and authenticate (where Google trusts you). Google redirects to http://localhost:PORT/callback?code=....
  4. Your phone has nothing on localhost:PORT → connection refused → the URL with a fresh, unconsumed code just sits in your address bar. You copy it and reply.
  5. The agent navigates its own local browser to that exact callback URL. Right session cookie + matching state + unconsumed code → the app completes the exchange → agent is authenticated. ✅
The non-obvious key

The redirect failing on your phone is the feature — it leaves the one-time code unspent for the agent to replay.

Counterintuitive: don't use tailscale serve here

If you used tailscale serve for the redirect URI instead of localhost, the handoff would break. The redirect to https://your-mac.ts.net/callback would actually resolve and complete on your Mac — but carrying your phone's session (no app cookie/state) — consuming the one-time code with the wrong session. For the handoff pattern, localhost beats serve.

Gotchas

  • Codes are single-use and short-lived (seconds–minutes). Be quick.
  • The agent — not your phone — must be the one to hit the callback.
  • Only works cleanly when agent-browser is local (so its localhost is where the app runs).

This is more fiddly than session reuse, so treat it as the “I really need a human-fresh login each time” path.

4. Where Tailscale actually fits

Your ts function (tailscale serve, in zsh/.config/zsh/tailscale.zsh) is the right tool for reachability, in three cases:

SituationToolNotes
Provider refuses http://localhost and demands HTTPS tailscale serve Register https://your-mac.ts.net/callback, point ts 3000 at the app, and do session reuse (not handoff) against it. Google itself allows loopback; many others don't.
A remote agent-browser on your tailnet must reach your local dev app tailscale serve Gives it a stable HTTPS hostname.
A cloud-provider browser (-p browserbase/kernel/...) not on your tailnet tailscale funnel Public internet exposure. Use dedicated test data and a hard-to-guess path.
Bottom line

Tailscale solves “how does this browser reach my app over a real URL,” never “how does the agent get past Google's login.”

5. Recommendation

For local agent-browser testing of OAuth apps, in order:

  1. --auto-connect state save → --state auth.json (or --profile). Robust, repeatable, no password exposure, no copying. This is what I'd set up.
  2. Headed local takeover when a session expires — type creds in the same window, agent resumes.
  3. The Telegram handoff as the escape hatch for mandatory-fresh-login flows, with a localhost redirect.
  4. Tailscale only when a provider forces HTTPS redirects or a remote/tailnet browser must reach your local app.
Security

Prefer a dedicated test Google account over your primary, regardless of approach. Session reuse hands the agent a token, not your password — that's strictly better than feeding credentials into an automated login.
